Meal-Planner Safety Harness

High-Level Design  ·  An AI agent that lives inside a safety harness

The agent only proposes a meal plan. The harness alone decides what reaches the user. All safety lives in the gap between propose and apply — never inside the model.

1 · The real-world problem

People let AI plan what they eat — but a meal plan is acted on, not a throwaway chat reply. A fluent AI is not a safe AI: the same model that writes a great plan will, just as confidently, write a dangerous one.

A confident AI can output……and a real person is harmed
Peanut butter for a peanut-allergic userAnaphylaxis
A 700 kcal/day “weight-loss” planStarvation-level diet, medically unsafe
An ingredient nobody confirmed you ownA plan you literally cannot cook

You cannot trust the model to police itself. The component generating the plan must not be the final word on whether it is safe to serve.

2 · How we address it — propose vs. apply

We don’t build a “safer model.” We wrap any model in a harness that owns every safety decision. The agent is powerful but untrusted: it makes proposals; nothing it says is authoritative until the harness clears it.

4 · The four pillars in one glance

PillarRoleKey property
🛡️ Guardrails
guardrails.py		Declare what “safe” meansEvery rule is a data object in one REGISTRY list — no safety logic hidden in the model or orchestrator. Tuning safety = editing the list.
✅ Checkpoints
checkpoints.py		Gate the proposal, in orderExplicit PASS/FAIL + the evidence judged on. SAFETY runs before anything can ship; a CRITICAL failure stops the line.
📦 Material
material.py		Move request in / proposal outTyped & immutable. The agent never touches disk or raw input — that decoupling is exactly what makes it swappable.
🚨 Alarms
alarms.py		Signal what went wrongStructured objects (type · severity · context · recommended action), streamed as JSONL — not log lines. CRITICAL alarms are blocking.

The declared guardrail registry — the entire safety surface, in one editable list

GuardrailSeverityRule
ALLERGEN_EXCLUSIONCRITICALNo ingredient matching a user allergen (alias-expanded: “peanut” → groundnut, arachis…) may appear.
CALORIE_FLOORCRITICALNo day, and no stated target, may fall below the 1200 kcal safety floor.
CALORIE_CEILINGWARNINGNo day may exceed the 5000 kcal sanity ceiling.
INGREDIENT_CONFIRMEDWARNINGEvery ingredient is pantry-stocked or explicitly flagged buy:.
DIET_COMPLIANCEWARNINGAll meals respect the declared diet type (vegan, pescatarian…).
BUDGET_LIMITWARNINGTotal plan cost stays within the weekly budget.

A failed guardrail returns structured violations, a worker-readable feedback string, and an auto_recoverable flag — the line between “retry the worker” and “stop and ask a human.”

5 · How the agent works & communicates

The agent never gets the last word, and never works blind. Each failed gate hands back a structured, worker-readable feedback string. Feedback is cumulative — the harness remembers every correction, so the agent converges instead of trading one defect for another.

Attempt 1
Harness → Agent: propose(request, feedback="")
Agent → Harness: plan with peanut butter + an unflagged ingredient
CP2 SAFETY — allergen → line STOPS
later gates never run on an unsafe plan
Attempt 2
Harness → Agent: propose(request, "remove peanut…")
Agent → Harness: peanut fixed, ingredient still unflagged
CP4 — unconfirmed ingredient
Attempt 3
Harness → Agent: propose(request, "remove peanut… + flag buy:")
Agent → Harness: both defects fixed
ALL GATES PASS ✓ → SHIP
Non-recoverable
e.g. a user’s stated 700 kcal target — Harness → Human: EscalationPacket “approve override or reject?”
Human rejects → BLOCK
never auto-guesses a number

Core guarantee: the agent’s behavior changes in response to the harness — not the other way around.

6 · How the final outcome is produced

Every run ends in exactly one verdict. Safety gates stop the line: a CRITICAL failure means no later gate runs, so an unsafe plan can never be “saved” by a downstream check.

CP1→CP4
all gates pass
→
✅ SHIP
|
↩ any gate fails
accumulate feedback → re-propose while attempts remain
→
🙋 non-recoverable / budget out
escalate to the human gate
OutcomeMeaning
SHIPPEDCleared every gate (often after self-correction).
BLOCKED · pending humanNon-recoverable safety issue, no human wired in → safe default: halt.
BLOCKED · rejectedA human reviewed and declined the override.
BLOCKED · loop exhaustedAgent couldn’t satisfy constraints in budget → escalates; never ships a guess.

Every checkpoint result is persisted to disk, so any run can be replayed from any checkpoint forward without re-invoking the agent.

7 · Technology — how it’s built

LayerStack
FrontendSingle static HTML page + vanilla JS (zero build)
BackendPython 3.12 · FastAPI · Uvicorn · Pydantic
Safety corePure Python, zero third-party deps — runs & is testable with no API key
Live agentsClaude sonnet-4-6 · Gemini 2.5-flash (lazy-imported)
Offline agentsMock + Template — deterministic, demo without keys
DeployGitHub → Railway (Nixpacks · Procfile · env-var keys)

Why this shape

  • SDKs are imported lazily → the whole safety engine runs offline; judges see it work with no key.
  • Workers are drop-in: the same harness drives Claude, Gemini, or a mock with zero harness changes.
  • API keys live only in the server environment — never in code or git.
  • The web layer is a thin shell: validate → pick worker → Harness.run → report. All safety stays in the harness.

The one-line pitch
Don’t make the model safe — put the model in a harness that’s safe. The agent proposes; declared guardrails, ordered checkpoints, structured alarms, and a stop-and-ask-a-human gate decide. Swap the model freely; the guarantees never move.
